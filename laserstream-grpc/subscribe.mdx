---
title: "Subscribe"
description: "Subscribe to accounts, transactions, blocks, and slot updates to receive real-time notifications using the high-performance gRPC API."
---

## Endpoints

gRPC services are available on mainnet and devnet with the following URLs:

- **Mainnet** `https://laserstream-mainnet.helius-rpc.com:443`
- **Devnet** `https://laserstream-devnet.helius-rpc.com:443`

## Authorizations

<ParamField query="api-key" type="string" required>
  Your Helius API key. You can get one for free in the [dashboard](https://dashboard.helius.dev/api-keys).
</ParamField>

## Body

The gRPC API supports multiple subscription types that can be combined in a single request:

<ParamField body="accounts" type="object">
  Subscribe to account updates. Returns data when specified accounts are modified.
  
  - `account`: Array of account pubkeys to monitor
  - `owner`: Array of owner pubkeys to monitor (all accounts owned by these programs)
  - `filters`: Optional filters to apply:
    - `memcmp`: Filter by specific bytes at an offset in account data:
      - `offset`: Position in bytes to start comparing data
      - One of: `bytes`, `base58`, or `base64`: The data to compare against
    - `datasize`: Filter by exact account data size in bytes
    - `token_account_state`: Filter to only token accounts
    - `lamports`: Filter by SOL balance with one of these comparisons:
      - `eq`: Equal to specified amount
      - `ne`: Not equal to specified amount
      - `lt`: Less than specified amount
      - `gt`: Greater than specified amount
  - `nonempty_txn_signature`: Include transaction signatures when available
</ParamField>

<ParamField body="slots" type="object">
  Subscribe to slot updates.
  
  - `filter_by_commitment`: Filter slots by commitment level
  - `interslot_updates`: Include intermediate slot status updates (PROCESSED, CONFIRMED, FINALIZED, FIRST_SHRED_RECEIVED, COMPLETED, CREATED_BANK, DEAD)
</ParamField>

<ParamField body="transactions" type="object">
  Subscribe to transaction updates.
  
  - `vote`: Include vote transactions
  - `failed`: Include failed transactions
  - `signature`: Monitor specific transaction signature
  - `account_include`: Only include transactions affecting these accounts
  - `account_exclude`: Exclude transactions affecting these accounts
  - `account_required`: Transactions must affect all of these accounts
</ParamField>

<ParamField body="transactions_status" type="object">
  Subscribe to transaction status updates.
  
  - `vote`: Include vote transactions
  - `failed`: Include failed transactions
  - `signature`: Monitor specific transaction signature
  - `account_include`: Only include transactions affecting these accounts
  - `account_exclude`: Exclude transactions affecting these accounts
  - `account_required`: Transactions must affect all of these accounts
</ParamField>

<ParamField body="blocks" type="object">
  Subscribe to block updates.
  
  - `account_include`: Only include blocks with transactions affecting these accounts
  - `include_transactions`: Include full transaction details
  - `include_accounts`: Include account updates
  - `include_entries`: Include block entries
</ParamField>

<ParamField body="blocks_meta" type="object">
  Subscribe to block metadata updates (lighter than full block updates).
  
  Returns block metadata including:
  - Slot number
  - Blockhash
  - Rewards
  - Block time
  - Block height
  - Parent slot and blockhash
  - Transaction count
  - Entry count
</ParamField>

<ParamField body="entry" type="object">
  Subscribe to entry updates.
  
  Returns entry data including:
  - Slot number
  - Entry index
  - Number of hashes
  - Hash
  - Transaction count
  - Starting transaction index
</ParamField>

<ParamField body="commitment" type="enum">
  The commitment level for the subscription:
  - `PROCESSED` (0): Processed by the current node
  - `CONFIRMED` (1): Confirmed by supermajority of the cluster
  - `FINALIZED` (2): Finalized by the cluster
</ParamField>

<ParamField body="accounts_data_slice" type="array">
  Specifies partial account data to receive:
  - `offset`: Position in bytes to start reading data
  - `length`: Number of bytes to read
</ParamField>

<ParamField body="ping" type="object">
  Enable ping-pong messages for connection health monitoring.
  - `id`: Numeric identifier for the ping
</ParamField>

<ParamField body="from_slot" type="integer">
  Starting slot to receive updates from. Updates for slots before this value will be excluded.
</ParamField>

<RequestExample>
```json Request
{
   "accounts":{},
   "slots":{},
   "transactions":{},
   "transactions_status":{},
   "blocks":{},
   "blocks_meta":{},
   "entry":{},
   "commitment":1,
   "accounts_data_slice":[
     {
       "offset": 0,
       "length": 64
     }
   ],
   "from_slot":1,
   "ping": {
     "id": 1
   }
}
```
</RequestExample>

## Commitment Levels

All subscriptions support the following commitment levels:

- `PROCESSED`: Processed by the current node (0)
- `CONFIRMED`: Confirmed by supermajority of the cluster (1)
- `FINALIZED`: Finalized by the cluster (2)

## Response Structure

Subscription responses include:

- `filters`: The filter names that matched this update
- One of the following update types:
  - `account`: Account data, owner, lamports, executable status, etc.
  - `slot`: Slot information and status updates
  - `transaction`: Full transaction details, signature, and metadata
  - `transaction_status`: Transaction execution status (success/error)
  - `block`: Complete block data with transactions, accounts, rewards, etc.
  - `block_meta`: Lightweight block metadata without full transaction details
  - `entry`: Entry details within a block
  - `ping`/`pong`: Connection health check messages
- `created_at`: Timestamp when the update was created
